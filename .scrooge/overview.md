Scrooge is a token-miserly coding agent that uses an expensive SOTA model (Scrooge) for planning/reviewing and a cheap model (Cratchit) for tool-heavy execution, connected via OpenRouter or an MCP plugin for Claude Code.

Architecture: The core is a tree-sitter code map (src/codemap.rs) that extracts symbols and call graphs deterministically — zero LLM tokens spent on understanding code structure. Scrooge receives compact briefs (symbol map + best-practice sections matching task keywords) and may make up to 3 `web_answer` lookups (Brave summarizer) per planning turn to settle library or API questions before committing to a plan — he never touches files or the shell. Cratchit executes tasks via a tool loop (src/tools.rs) with file editing, shell, python/wolfram computation, docs queries, and call-graph navigation — all compressed into ≤6-line reports back to Scrooge. The orchestrator can selectively withhold tools from Cratchit per task via `cratchit_execute_with` (e.g., `write_file` is withheld during `ensure_overview` so the orchestrator, not Cratchit, saves overview.md). Key invariants: (1) Every LLM call is logged to `.scrooge/accounts.log` (prompt text and output text only — tool-call messages are filtered out) and aggregated per-agent in `.scrooge/ledger.json`, which also records a `shillings_saved` USD figure computed from `.scrooge/rates.toml` (Scrooge-model token prices); all best-effort, never failing the call. (2) In plugin mode, Claude plays Scrooge while the binary serves tools over MCP (src/mcp.rs); hooks inject context at session start and after user prompts. (3) The post-task check suite (src/checks.rs) runs formatters, then tests, then lint autofix — tests must pass before lint warnings are surfaced. (4) Helper inventory (src/helpers.rs) uses heuristics (fan-in ≥2, utility-shaped names, ≤100 lines) to find reusable utilities, optionally validated by Cratchit and cached. (5) The `.scrooge/` directory holds mutable state (checks.toml, rates.toml, ledger.json, accounts.log, overview.md, helpers.json) that agents may edit; the orchestrator manages creation of overview.md to prevent Cratchit from writing it prematurely.
