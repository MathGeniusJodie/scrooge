Scrooge is a token‑miserly two‑tier coding agent where an expensive SOTA model (Scrooge) plans from compact briefs while a cheap model (Cratchit) executes all file‑reading, code‑writing, and verification with a full tool belt.

The architecture is built around a single invariant: never burn LLM tokens on work a deterministic program can do. A tree‑sitter‑based code map (`codemap.rs`) extracts every symbol and the call graph at zero token cost, with git‑aware file filtering (`git ls-files`) ensuring ignored/generated files never enter any scan; an in‑process mtime+count cache (`build_cached`) avoids redundant re‑parsing within a session. HTML files are parsed for `<script>` bodies and indexed as JavaScript, and a 262KB file‑size cap plus an expanded skip list keep scans fast. A keyword‑matched best‑practices document (`practices.rs`) injects only relevant guidance, keyed to the project's actual languages via `CodeMap::languages()`. A reusable‑helper index (`helpers.rs`) prevents reinventing utilities. The `brief_for` method slices the codebase brief to show full listings only for task‑relevant files/symbols, and `structure_signature` produces a line‑number‑exclusion fingerprint that detects when the project overview has gone stale. The orchestration loop (`agents.rs`) chains: deterministic context gathering → one Scrooge planning turn (with a KV‑cache‑friendly prefix rebuilt once per task) → Cratchit execution → mechanical check verification (`checks.rs`), with check failures retrying straight through Cratchit in the same live conversation to avoid burning expensive Scrooge rounds. Scrooge is only told DONE when checks pass green. The `Chat` trait accepts a separate `reasoning_max_tokens` so Scrooge's thinking budget (capped at 1024 tokens) is isolated from visible output (2048 tokens) — a long think can't starve the tool call. When output is clipped (`finish_reason: length`) with no tool calls, the orchestrator surfaces an explicit error rather than silently treating the truncated reply as a successful finish. Scrooge's direct `read_file` calls are also clamped to a 50‑line peek, steering him toward the cheaper call‑graph tools (`symbol_info` / `read_symbol`) instead of paging through source. A `syntax_error_line` check in the code map lets write/edit tools flag broken edits in the same turn without waiting for the full test run. Overview generation is deferred for fresh projects until code exists — `ensure_overview` returns an empty string when the tree is empty, and `refresh_overview` handles first‑time generation after the task completes.

A dedicated cleanup module (`cleanup.rs`) runs the full check suite, applies mechanical autofixes, then splits the remaining diagnostics per‑language (clippy, ruff, biome, structure) and hands each problem to Cratchit one at a time, re‑verifying after every fix — keeping each delegation small and targeted. The same binary serves dual roles: a standalone CLI with subcommands (`run`, `cratchit`, `ask`, `map`, `check`, `cleanup`, `helpers`, …) and an MCP stdio server (`mcp.rs`) where a Claude Code conversation plays Scrooge and delegates legwork to Cratchit. The MCP server exposes the cleanup tool directly so the conversation can trigger the full fix‑loop without burning its own tokens. A Claude Code plugin hooks into the stop event and calls `scrooge cleanup` plus `scrooge refresh‑overview` to ensure the tree is clean after every session, retrying up to five times before reporting failure.

A project overview (`.scrooge/overview.md`), written by Cratchit on first use and refreshed after structural changes (detected via `CodeMap::structure_signature`), is injected verbatim into every briefing so both agents share the same architectural understanding. Every LLM call is bookkept in `accounting.rs` with per‑agent token counts and a "shillings saved" metric computed against configurable Scrooge‑model rates (`.scrooge/rates.toml`), making the economic thesis of the two‑tier split visible in `.scrooge/ledger.json`.
